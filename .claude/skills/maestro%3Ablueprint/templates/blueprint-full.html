Blueprint: WebSocket Real-Time Events

Replace HTTP polling with persistent WebSocket connections for sub-100ms event delivery

1 Executive Summary
12
Files Changed
3

~800
Lines of Code
Medium
Risk Level
The editor currently uses HTTP polling at 3-second intervals to sync changes between users. This creates noticeable lag (2-5 seconds) and wastes bandwidth with empty poll responses (>90% of requests return no changes). WebSocket connections will deliver events in under 100ms while eliminating unnecessary network traffic.

3 Phased Implementation
Phase 1 Core WebSocket Layer ~3 days
WS handler, connection lifecycle, auth integration, and heartbeat. The server accepts connections and can push messages to connected clients.
Phase 2 Event Bus & Subscriptions ~2 days
Pub/sub event bus, topic-based subscriptions, and service integration. Events flow from services through the bus to connected clients.
Phase 3 Client Migration & Cleanup ~2 days
Replace client-side polling with WS subscription, add reconnection logic, remove polling infrastructure, and verify end-to-end latency.

6 Risk Matrix
RiskImpactLikelihoodMitigation
Connection storms on deploy (all clients reconnect simultaneously) High Medium Exponential backoff with jitter on client reconnect. Rate-limit WS upgrades per IP.
Memory leak from uncleaned stale connections Medium Low Heartbeat mechanism (Phase 1) cleans up within 60s. Add connection count metrics.
Breaking change in client sync during migration High Low Keep polling endpoint alive during Phase 3. Feature flag to toggle WS vs polling per user. Remove polling only after 1 week of stable WS.
Event bus becomes bottleneck under high write load Medium Low In-process pub/sub is fast for current scale (~100 concurrent). Monitor event throughput. Redis pub/sub is the escape hatch if needed later.
