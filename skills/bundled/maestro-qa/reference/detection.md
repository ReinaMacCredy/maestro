# Detection rules

The installer does its analysis pass before asking any question. This file is the rule set.

## Modality detection

Run all rules; record every modality whose strong signals match. A repo can have multiple modalities.

### `cli`

Strong signals:

- `package.json` has a `bin` field.
- `Cargo.toml` declares `[[bin]]`.
- `go.mod` plus a top-level `main.go` or `cmd/<name>/main.go`.
- `pyproject.toml` declares `[project.scripts]` or `[tool.poetry.scripts]`.
- A `Makefile` `install:` target that copies an executable to `$(PREFIX)/bin`.

### `web`

Strong signals:

- Dependency on any of: `react`, `next`, `vue`, `nuxt`, `svelte`, `@sveltejs/kit`, `@angular/core`, `solid-js`, `astro`, `remix`, `gatsby`.
- Top-level `index.html` or `public/index.html`.
- `vite.config.*`, `webpack.config.*`, `next.config.*`, `astro.config.*`.

### `api`

Strong signals:

- Dependency or import of: `express`, `fastify`, `koa`, `@nestjs/core`, `hapi`, `fastapi`, `flask`, `django`, `rails`, `sinatra`, `phoenix`, `gin-gonic/gin`, `axum`, `actix-web`, `tide`.
- A `routes/` or `controllers/` directory at a top level.
- An `openapi.yaml`, `swagger.yaml`, or `*.openapi.json`.

### `mobile`

Strong signals:

- `Package.swift` with an iOS/iPadOS/tvOS/watchOS target.
- `android/build.gradle` with `com.android.application` plugin.
- Dependency on `react-native` or an `expo` config.
- Flutter `pubspec.yaml` with `flutter:` block.

### `desktop`

Strong signals:

- Dependency on `electron`, `@electron/forge`, or `@electron-forge/cli`.
- `src-tauri/` directory or `tauri.conf.json`.
- `Package.swift` with a macOS-only target (no iOS/iPadOS).
- `.csproj` referencing `Microsoft.WindowsDesktop.App` or `WPF`/`WindowsAppSDK`.
- GTK/Qt manifest (`.desktop`, `*.pro`, `meson.build` with `gtk`/`qt`).

### `tui`

Strong signals:

- Dependency on `@opentui/core`, `ink`, `blessed`, `react-blessed`, `textual`, `urwid`, or `github.com/charmbracelet/bubbletea`.
- A repo command of the shape `<bin> ... --render-check ...` (grep `--render-check` across the source tree).

## Integration detection

Look for imports, env files, and config:

| Integration | Signals |
|---|---|
| Postgres | `pg`, `postgres`, `psycopg`, `DATABASE_URL` referencing `postgres://`. |
| MySQL | `mysql2`, `mysqlclient`, `DATABASE_URL` referencing `mysql://`. |
| SQLite | `better-sqlite3`, `sqlite3`, `*.db` in source. |
| Redis | `redis`, `ioredis`, `REDIS_URL`. |
| Mongo | `mongodb`, `mongoose`, `MONGO_URL` / `MONGODB_URI`. |
| S3-compatible | `aws-sdk`, `@aws-sdk/client-s3`, `boto3`, `S3_BUCKET`, `MINIO_*`. |
| Stripe | `stripe`, `STRIPE_SECRET_KEY`. |
| OpenAI / Anthropic | `openai`, `@anthropic-ai/sdk`, `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`. |
| GitHub API | `@octokit/*`, `GH_TOKEN`, `GITHUB_TOKEN`. |
| Generic queue | `bullmq`, `bee-queue`, `kafkajs`, `nats`, `rabbitmq`, `amqplib`. |
| Generic HTTP API | env vars matching `*_API_URL` / `*_BASE_URL` not covered above. |

For each match, record `{name, kind, env_vars}`. The user picks coverage in questionnaire category 3.

## CI provider detection

| Provider | Signal |
|---|---|
| GitHub Actions | `.github/workflows/*.yml` |
| GitLab CI | `.gitlab-ci.yml` |
| CircleCI | `.circleci/config.yml` |
| Buildkite | `.buildkite/pipeline.yml` |
| Azure Pipelines | `azure-pipelines.yml` |
| Jenkins | `Jenkinsfile` |

Only `github-actions` is generated by the installer in v1; other providers are recorded in the report so the user knows to wire up `/qa` themselves.

## Existing tests detection

Walk these dirs if present and record framework-config files: `tests/`, `test/`, `__tests__/`, `spec/`, `e2e/`. Record any of: `vitest.config.*`, `jest.config.*`, `playwright.config.*`, `cypress.config.*`, `pytest.ini`, `pyproject.toml [tool.pytest.ini_options]`, `go.mod` with `_test.go` files, `Cargo.toml [[test]]`, `phpunit.xml`, `RSpec`.

The installer does not run them; their presence informs the questionnaire defaults (e.g., a repo with Playwright already configured nudges the `web` modality default to `agent-browser`).

## Critical-flow candidate detection

For each modality, generate up to 5 candidate flow names from the strongest evidence:

- `cli`: subcommands enumerated from the bin's `--help` output if runnable, else from a registry file (Commander/Click/cobra).
- `web`: top-level routes from `pages/`, `app/`, or a router config.
- `api`: endpoints from route files; method + path.
- `mobile`: top-level screens from a navigator config.
- `desktop`: window/menu names from main process source.
- `tui`: screens from `--render-check --screen <name>` enumeration if available.

Present these as defaults in questionnaire category 2; user keeps, edits, or replaces.

## App-slug rules

The slug is used as the directory name (`qa-<slug>`) and the key under `apps.<slug>` in `config.yaml`.

1. One app per modality is the common case. Slug = the modality name. Examples: `qa-cli`, `qa-web`, `qa-api`.
2. Multiple apps of the same modality (monorepo): slug = `<modality>-<package-folder>`. Examples: `qa-web-marketing`, `qa-web-admin`, `qa-api-billing`.
3. Slugs are lowercased, kebab-cased, ASCII only, max 40 characters. Strip leading/trailing dashes.
4. If a derived slug collides with an existing one, append a numeric suffix: `qa-web-2`. The installer warns when a numeric suffix is required so the user can pick a better folder name in the manifest.
